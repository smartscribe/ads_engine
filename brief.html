JotPsych — Ads Engine Project Brief

Ads Engine

We're building a system to run JotPsych's paid ads like a quant trading desk: generate creative, deploy it, measure what works, kill what doesn't, and feed the learnings back into the next round. Nate started the Claude Code harness — data models, pipeline architecture, decision logic, regression framework — but it is not operational. That's where you come in.

Every cycle makes the next batch of creative smarter. The system gets better the more you use it.

Here's what you need to know

JotPsych is an AI documentation tool for behavioral health clinicians. We run paid ads on Meta and Google to acquire users. The primary metric is cost per first note completion (CpFN). We spend ~$15–20K/mo and need to scale.

This project is for both intern candidates (as a paid evaluation exercise) and current team members picking up the work. Either way, the expectations are the same.

Here's what we're going to ask of you

  1. Build the image and video generation pipeline — copy generation works, you solve visual assets at scale
  2. Wire up Meta and Google APIs — deploy ads programmatically, pull performance data back daily
  3. Validate the regression model — our creative taxonomy needs real data; confirm the coefficients are orthogonal
  4. Ship the review dashboard — gallery UI for Nate and Jackson to approve/reject variants
  5. Connect Slack notifications — daily digest of scale/kill/wait decisions, deployment alerts

This exercise is about quality at speed

We care about velocity — the whole point of this system is to generate and test creative faster than a human team ever could. But speed without quality is just burning money faster. Three kinds of quality, specifically:

  • Data quality. If the numbers going into the regression are wrong, the coefficients are meaningless. Attribution, conversion tracking, spend data — all of it has to be right before anything else matters.
  • Insight quality. The regression model should tell us something we didn't already know. "Video outperforms static" is obvious. "Audit-risk framing with a specific number in the headline drives 40% lower CPA than pain-point framing" is an insight. Get to the second kind.
  • Creative output quality. The ads this system produces cannot look or read like AI generated them. If a clinician scrolling Instagram can tell it's AI slop, it's worthless. The quality bar is: would a human creative director be proud of this?

The goal is high-quality creative at a pace that would be impossible manually. Both matter. Don't sacrifice one for the other.

Open Questions

Priority 1 — Creative Generation

Copy generation works. Visual asset generation is the hardest unsolved problem. Don't assume you need fancy tooling — one viable approach is just Claude with good skills: generate HTML/CSS ads, use Playwright to screenshot them, composite with stock imagery. Caveman-style, but it ships fast and you control every pixel. Evaluate that alongside the more exotic options.

  • 1What's the generation approach? Options range from simple (Claude skills generating HTML/CSS → screenshot → composite) to complex (Flux, DALL-E, Ideogram APIs). Can Claude Code with Playwright handle static ad creation directly? What about simple motion — CSS animations screencaptured as video? Start with what's fast and controllable, then graduate if needed.
  • 2Can we do video? Evaluate Runway, Pika, Kling for generated video. But also evaluate simpler paths: animated HTML captured via Playwright, screen recordings of the product, motion graphics from templates. Even 5-second loops add value for Reels/Stories.
  • 3How do we prevent AI slop? Curated style references, negative prompts, post-processing, quality filters. If a clinician can tell it's AI, it's worthless. The Claude-skills approach may actually be better here — you're composing design elements, not generating pixels.
  • 4Brand consistency. How do we keep visual identity coherent across variants? Style guides? LoRA fine-tunes? Template overlays? Design tokens in CSS?
  • 5Asset specs. Each platform + placement has different aspect ratios, file sizes, text overlay limits. Build the spec mapping.

Priority 2 — Platform Integration

Full API integration with Meta and Google. Deploy from the harness, pull metrics back.

Meta Setup

Business Manager, App with ads_management permission, long-lived access token, ad account linked.

Google Setup

Developer token application, OAuth flow, customer ID linking. Document the process.

Campaign Structure

How to organize campaigns / adsets / adgroups? Per theme? Per audience? Per format? This affects measurement.

Conversion Tracking

Confirm pixel/tag setup for first-note-completion events. The Google tag is currently broken. If tracking isn't solid, nothing else matters.

Priority 3 — Data & Analytics

The regression model decomposes creative elements into coefficients. It tells us what actually drives conversions — but only if the data going in is clean.

  • 10Is the MECE taxonomy right? Run the first batch through it. Are the categories exhaustive? Overlapping? Check VIF scores once you have data.
  • 11When does the regression become useful? Estimate: how many variants x how many days before coefficients are meaningful?
  • 12Data pipeline. Current store is JSON files. At what point do we need SQLite or Postgres? Build the migration path.

Priority 4 — Process & UX

  • 13Dashboard. The review gallery is an HTML prototype. Should this become a React app? Your call, but it needs to feel fast and look good.
  • 14Scheduler. The daily cycle needs to run automatically. Cron? Cloud function? Pick the simplest thing that works.
  • 15Feedback loop. How do rejection notes from review feed back into the generator? Few-shot examples? Prompt engineering?

Quickstart

How to Get Started

Clone the repo, create your own copy, and push regularly. We want to see your work as it progresses — not just a final deliverable.

# 1. Clone the repo and create your own version
git clone https://github.com/smartscribe/ads_engine.git ads_engine_yourname
cd ads_engine_yourname

# 2. Set up your own remote (create a repo on GitHub first)
git remote set-url origin https://github.com/yourname/ads_engine.git
git push -u origin main

# 3. Set up environment
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 4. Copy and fill in secrets
cp .env.example .env

# 5. Test the intake + generation pipeline (needs ANTHROPIC_API_KEY)
python -m engine.orchestrator idea "Therapists spend 2 hours \
on notes after every session. What if they didn't have to?"

# 6. Start the dashboard API
uvicorn dashboard.api.app:app --reload

# 7. Open the review gallery
open dashboard/frontend/pages/review.html

Push to git regularly. Commit often, with clear messages. We review your commit history — it tells us how you think, not just what you built. Small, working commits are better than one big dump at the end.
