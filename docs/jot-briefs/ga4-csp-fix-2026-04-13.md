# GA4 Tracking Investigation and Fix — jotpsych.com CSP Bug

**Date:** 2026-04-13
**Author:** Nate + Claude
**Affected surface:** jotpsych.com marketing site (Netlify-hosted)
**Severity:** High — GA4 pageview data has been missing or dramatically undercounted for an unknown period (at least weeks, possibly months).
**File changed:** `jotpsych_gtm/new_landing_page/site/_headers`
**Deployment status at time of writing:** Edit staged on disk, not yet deployed. Ships via `/push-to-jotpsych.com`.

---

## TL;DR

Google Analytics for jotpsych.com has been broken. The Content Security Policy header served by Netlify was too restrictive for modern GA4 traffic patterns, and every GA4 measurement hit from the marketing site has been silently killed by the browser at the CSP layer before leaving the page. The tag libraries loaded, so it *looked* installed, but none of the actual data ever reached Google. This went unnoticed in part because a second GA4 property (mislabeled "smartscribe.health") was still showing healthy volume — but it turned out that property was receiving its data from `app.jotpsych.com`, which has a looser CSP, not from the marketing site. Both GA4 properties have effectively been tracking only app activity, not marketing-site activity, for the duration of this bug. The fix is a one-line additive change to the Netlify `_headers` file to whitelist the Google domains GA4 actually uses. No existing security restriction is weakened.

---

## Symptom That Kicked This Off

Nate opened the GA4 property he believed was tracking jotpsych.com and saw roughly **200 new users in the last 30 days**. That number is an order of magnitude too low for a site running $15–20k/mo in paid media. His first instinct was that Google Tag Manager might be broken, and that turned out to be correct in substance, though the actual mechanism was one level deeper than a simple GTM misconfiguration.

## Diagnostic Steps Taken

### Step 1: Identify which GA4 property actually owns jotpsych.com

Nate had many GA4 properties in his account and was not sure which one corresponded to jotpsych.com. Rather than click through them manually, I fetched the live jotpsych.com homepage and pulled the GTM container ID from the page source:

```
GTM-KL9RPN9V
```

I then fetched the GTM container configuration directly from `googletagmanager.com/gtm.js?id=GTM-KL9RPN9V` and extracted all GA4 measurement IDs it was firing. The container turned out to be **dual-writing to two separate GA4 properties**:

| Measurement ID | GA4 Property | Data stream label | URL label |
|---|---|---|---|
| `G-B7Q5FRBQRH` | `419769857` | `www.jotpsych.com` | `www.jotpsych.com` |
| `G-PY509QJK6Z` | `421925451` | `Web app` | `https://smartscribe.health` |

The 200 new users number Nate had originally seen was actually from the second property (`421925451`), not the first — but the fact that it was that low for a supposedly dead domain was itself confusing. Jackson confirmed that nobody uses smartscribe.health anymore and everyone is supposed to be on jotpsych.com, including the app itself at `app.jotpsych.com`.

### Step 2: Verify whether GA4 is actually firing on jotpsych.com

To verify whether GTM and GA4 were actually functioning, I loaded the live site in a headless Playwright browser and captured every network request. Results:

| Request | Status | Meaning |
|---|---|---|
| `googletagmanager.com/gtm.js?id=GTM-KL9RPN9V` | 200 OK | GTM container loaded |
| `googletagmanager.com/gtag/js?id=G-B7Q5FRBQRH` | 200 OK | First GA4 tag library loaded |
| `googletagmanager.com/gtag/js?id=G-PY509QJK6Z` | 200 OK | Second GA4 tag library loaded |
| `google.com/ccm/collect?en=page_view` (×2) | **FAILED — `ERR_BLOCKED_BY_ORB`** | Google Ads conversion linker hit — blocked |
| `google-analytics.com/g/collect` | **NEVER FIRED** | **This is the critical one. The actual GA4 measurement hit that delivers pageview data never made it to the network.** |

That last observation is the smoking gun. The tag libraries were loading fine, but the real measurement events that send data to Google were never making it out of the browser. Without those hits, GA4 gets nothing, and the property shows nothing.

### Step 3: Cross-check with Google Tag Assistant

To rule out a headless-browser artifact, Nate opened jotpsych.com in his real Chrome with DevTools and confirmed the same absence of `/g/collect` requests, then installed Google's official Tag Assistant Chrome extension to get a verdict from Google's own tooling. Tag Assistant returned a definitive diagnosis:

> **Content Security Policy (CSP) blocking Google scripts.** The page's security settings (Content Security Policy) are preventing scripts from Google domains like `googletagmanager.com` and `google-analytics.com` from loading or executing. This issue needs to be resolved by a web developer or server administrator. They will need to update the website's `Content-Security-Policy` HTTP header to include the necessary Google domains in the `script-src-elem`, `img-src`, and `connect-src` directives.

Three independent sources — headless browser capture, real browser capture, and Google's own Tag Assistant — all pointed at CSP as the culprit.

### Step 4: Pull the live CSP header

```bash
curl -sI https://jotpsych.com | grep -i content-security-policy
```

Result (broken into readable chunks):

```
default-src 'self';
script-src  'self' 'unsafe-inline' https://www.googletagmanager.com https://api.hsforms.com;
style-src   'self' 'unsafe-inline' https://fonts.googleapis.com;
font-src    'self' https://fonts.gstatic.com;
img-src     'self' https: data:;
connect-src 'self' https://api.hsforms.com https://www.google-analytics.com;
frame-src   https://www.googletagmanager.com https://open.spotify.com
```

On the surface this looks fine — `connect-src` does whitelist `www.google-analytics.com`. But this is where the subtle problem lives.

## Root Cause

**Modern GA4 does not send its telemetry to `www.google-analytics.com` in most cases.** It uses regional endpoints like `region1.google-analytics.com`, `region2.google-analytics.com`, and so on, plus `analytics.google.com` for some hit types. The bare `www.` subdomain in the CSP does not match those regional hosts, so every GA4 hit routed to a regional endpoint was being silently killed at the CSP layer before it could leave the user's browser. Additional problems in the existing CSP:

| Directive | Missing entry | Consequence |
|---|---|---|
| `connect-src` | `https://*.google-analytics.com` | GA4 regional endpoints (`region1.`, `region2.`, etc.) blocked. **Main killer.** |
| `connect-src` | `https://*.analytics.google.com` | Alternate GA4 ingestion hostname blocked. |
| `connect-src` | `https://www.google.com` | Google Ads conversion linker `/ccm/collect` blocked. |
| `connect-src` | `https://*.g.doubleclick.net` | Google Signals / remarketing pixels blocked. |
| `connect-src` | `https://server-side-tagging-z4zkeg2xnq-uc.a.run.app` | Server-side tagging Cloud Run endpoint blocked. |
| `frame-src` | `https://td.doubleclick.net`, `https://www.google.com` | Conversion linker iframe (`sw_iframe.html`) blocked. |
| `script-src` | `https://*.googletagmanager.com` (wildcard) | Functional but fragile — single literal subdomain only. |

**The net effect: the tag libraries loaded (because `script-src` allowed the JS files), but every actual measurement event that tried to send data back to Google was blocked by the browser before it left the page.** The tag was "installed but completely dead." Visitors to jotpsych.com were being tracked into a void.

## The smartscribe.health Property Confusion

This deserves its own section because it initially seemed to contradict the CSP diagnosis.

### The puzzle

If CSP is blocking all GA4 hits from jotpsych.com, then the second property (`421925451`, G-PY509QJK6Z) receiving hits via the GTM container's dual-write should show zero traffic — not "significant volume and looks healthy." So either the CSP theory was wrong, or the healthy data was coming from somewhere other than jotpsych.com.

### The investigation

I loaded `app.jotpsych.com` in Playwright the same way I had loaded jotpsych.com, and captured its network requests. The findings were definitive.

**First — app.jotpsych.com loads the exact same GTM container (`GTM-KL9RPN9V`) that jotpsych.com loads.** The marketing site and the app share one GTM setup.

**Second — on app.jotpsych.com, all the GA4 measurement hits succeeded** with HTTP 200 and 204 responses, unlike on jotpsych.com where they were blocked. The app's CSP is either much looser than the marketing site's, or absent entirely.

**Third — the routing of the two measurement IDs splits asymmetrically:**

| Measurement ID | Endpoint | Status on app.jotpsych.com |
|---|---|---|
| `G-PY509QJK6Z` (property `421925451`) | `analytics.google.com/g/collect` — direct to Google | 204 success |
| `G-B7Q5FRBQRH` (property `419769857`) | `server-side-tagging-z4zkeg2xnq-uc.a.run.app/g/collect` — Cloud Run server-side container | 200 success |
| Both (auxiliary) | `stats.g.doubleclick.net/g/collect`, `google.com/ccm/collect` | 204 / 200 success |

### Correction to an earlier concern

In my first writeup of the CSP fix I flagged that the server-side tagging Cloud Run endpoint had not been verified to still be alive and suggested someone should check it before whitelisting it. **That concern is now resolved with evidence.** The endpoint is confirmed alive and actively serving production traffic — it is receiving the `G-B7Q5FRBQRH` events from app.jotpsych.com in real time. Whitelisting it in the jotpsych.com CSP was the correct call.

**Both GA4 properties have been receiving only `app.jotpsych.com` data, not marketing-site data, for the entire duration of the CSP bug.** The marketing site has been completely untracked.

The property labeled "smartscribe.health" (`421925451`) looks healthy because `app.jotpsych.com` fires into it directly via `analytics.google.com`, bypassing the GTM container's CSP problem on jotpsych.com. The "Web app" data stream name and `https://smartscribe.health` URL label inside that property's Data Streams admin page are **stale legacy decoration** from when the app used to live at that domain. GA4 data streams collect based on the measurement ID embedded in the tag, not based on the URL label shown in the admin UI. When the app migrated from smartscribe.health to app.jotpsych.com, the same measurement ID kept firing (because nobody changed the frontend tag config), and the data kept flowing into the same property. Only the display label fell out of date.

**Implication:** the two properties are not currently split as "marketing" vs. "app" the way one might expect. Both are being written to from both surfaces (or will be, once the CSP fix ships). There is no clean separation between site funnel data and in-app behavior data. This is technical debt worth cleaning up but not urgent for the immediate goal of getting landing-page data flowing.

## Fix Applied

### Diff

**Before:**
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' https://www.googletagmanager.com https://api.hsforms.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' https: data:; connect-src 'self' https://api.hsforms.com https://www.google-analytics.com; frame-src https://www.googletagmanager.com https://open.spotify.com
```

**After:**
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' https://*.googletagmanager.com https://api.hsforms.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' https: data:; connect-src 'self' https://api.hsforms.com https://*.google-analytics.com https://*.analytics.google.com https://*.googletagmanager.com https://www.google.com https://*.g.doubleclick.net https://server-side-tagging-z4zkeg2xnq-uc.a.run.app; frame-src https://*.googletagmanager.com https://td.doubleclick.net https://www.google.com https://open.spotify.com
```

### Changes, itemized

| Directive | Action | Value |
|---|---|---|
| `script-src` | Replaced literal with wildcard | `https://www.googletagmanager.com` → `https://*.googletagmanager.com` |
| `connect-src` | Added | `https://*.google-analytics.com` |
| `connect-src` | Added | `https://*.analytics.google.com` |
| `connect-src` | Added | `https://*.googletagmanager.com` |
| `connect-src` | Added | `https://www.google.com` |
| `connect-src` | Added | `https://*.g.doubleclick.net` |
| `connect-src` | Added | `https://server-side-tagging-z4zkeg2xnq-uc.a.run.app` |
| `frame-src` | Replaced literal with wildcard | `https://www.googletagmanager.com` → `https://*.googletagmanager.com` |
| `frame-src` | Added | `https://td.doubleclick.net` |
| `frame-src` | Added | `https://www.google.com` |

### Safety properties

- **Strictly additive.** No existing directive was removed or weakened. `default-src 'self'`, `'unsafe-inline'`, `img-src 'self' https: data:`, `font-src`, `style-src`, and every other existing restriction remain in place.
- **No new XSS surface.** Only specific, named Google domains are permitted — not `*`, not any arbitrary third-party hostname.
- **Reversible.** One commit. Revert and redeploy = back to current state in ~2 minutes.
- **Atomic deploy.** Netlify serves the new header or the old one, never a mix.

## Deployment and Verification Plan

### Deployment

The edit is staged on disk in `jotpsych_gtm/new_landing_page/site/_headers`. To deploy:

```
/push-to-jotpsych.com
```

Or manually commit, push to GitHub, and let Netlify auto-deploy.

### Verification (run in order after deploy)

1. **Confirm the header is live.**
   ```bash
   curl -sI https://jotpsych.com | grep -i content-security-policy
   ```
   Should contain the wildcarded Google domains.

2. **Confirm GA4 hits fire in a real browser.** Open jotpsych.com in Chrome → DevTools → Network tab → filter `collect` → reload. Expect to see a request to `*.google-analytics.com/g/collect` with status 200 or 204 (not blocked).

3. **Re-run Google Tag Assistant** on jotpsych.com. The CSP warning should be gone.

4. **Confirm Realtime data lands in GA.** Open GA4 property `419769857` → Reports → Realtime. Load jotpsych.com in another tab, interact briefly. Expect to appear as an active user within ~30 seconds. **This is the definitive proof that the pipe is live.**

5. **Wait 24 hours.** Compare day-over-day sessions and new users. Expect a dramatic step-change upward. If not, the fix did not take effect or there is a second bug.

## What This Does Not Fix

- **The 2FA signup flow break.** SMS verification for signups disrupts the tag fire at the moment of conversion. That is a separate bug and needs its own investigation.
- **The Google tag conversion discrepancy** (2 conversions in-platform vs. 15 in Metabase UTMs). May be *partially* improved by this fix if CSP was blocking Google Ads conversion pixels alongside GA4 hits, but the 2FA issue almost certainly accounts for the bulk of that gap.
- **Historical data recovery is not possible.** The months of missing GA4 data during the CSP bug are gone and cannot be backfilled. All meaningful trend analysis should start from the deploy date of this fix.
- **Post-fix numbers may be jarring.** Expect much higher than the current 200/month baseline, but potentially smaller than vanity-metric expectations. A clean baseline is better than a broken one regardless.

## Remaining Open Questions

1. **Is cross-domain tracking configured in the GTM container?** A user who lands from an ad on jotpsych.com and clicks through to app.jotpsych.com should appear as one continuous session with the ad as the source. This requires a GTM audit to confirm the Google tag has a linker config including both `jotpsych.com` and `app.jotpsych.com`. If not, the shared GTM container is not delivering its main benefit.

2. **Should the dual-write to two GA4 properties be collapsed?** Writing every event to both `419769857` and `421925451` doubles data volume and creates reporting ambiguity. A collapse to single-property is cleaner. `419769857` is the natural choice since it is explicitly labeled for the production `www.jotpsych.com` stream. But this is a reporting-continuity decision, not a pure engineering one — any historical trend lines in `421925451` need to be preserved or migrated first.

3. **Does `421925451` still need to exist at all?** If the dual-write is removed and it stops receiving new data, it becomes an abandoned archive. Decide whether to keep for historical continuity, migrate data out, or delete.

4. **Why does the app fire `G-PY509QJK6Z` directly to `analytics.google.com` but route `G-B7Q5FRBQRH` through the Cloud Run server-side container?** That asymmetric routing suggests someone set up server-side tagging for one measurement ID but not the other — possibly as an A/B comparison, possibly as an incremental migration. Worth understanding that history before making further changes.

5. **What is the `app.jotpsych.com` CSP policy?** Empirically it permits all the Google domains, but it should be audited for the same reasons we audited jotpsych.com's CSP — to confirm it is correctly scoped and not over-permissive in some other way.

6. **Ads_engine GA4 integration.** The original task that kicked this investigation off was to wire up `engine/tracking/ga_tracker.py` to pull landing-page performance data from the Data API. That work is still pending and will happen once the CSP fix is deployed and verified, at which point there will actually be data in property `419769857` to pull.
